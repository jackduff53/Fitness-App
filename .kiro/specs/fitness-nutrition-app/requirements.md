# Requirements Document

## Introduction

A native iOS fitness and nutrition tracking application built with SwiftUI targeting iOS 26+. The app integrates with Apple HealthKit to read fitness data (steps, active energy, workouts synced from Garmin via Apple Health, and resting heart rate), provides a natural language food logging experience powered by Apple's on-device Foundation Models framework, tracks daily eating goals, and offers static rule-based workout recommendations. No external API keys or network connectivity is required for food parsing. The app displays 7-day trend sparklines for key metrics and streak counters for goal consistency. All data is persisted locally using SwiftData with no external databases or login flows. The UI follows a deep cool-dark mode design with cyan-accent colors and tab-based navigation.

## Glossary

- **App**: The iOS fitness and nutrition tracking application
- **Dashboard**: The home screen displaying summary cards and progress indicators for fitness and nutrition data
- **HealthKit_Service**: The component responsible for reading health data from Apple HealthKit
- **Food_Logger**: The component responsible for capturing, parsing, and storing food log entries
- **Food_Parser**: The component responsible for parsing natural language food descriptions using Apple's on-device Foundation Models framework to return structured nutritional data
- **Foundation_Models**: Apple's on-device language model framework available on iOS 26+ that enables local natural language processing without network connectivity
- **Goal_Tracker**: The component responsible for tracking daily calorie and macronutrient targets
- **Workout_Recommender**: The component responsible for generating static rule-based workout suggestions
- **Progress_Ring**: A circular visual indicator showing percentage completion of a metric
- **Progress_Bar**: A horizontal visual indicator showing percentage completion of a macronutrient target
- **Net_Calorie_Balance**: The difference between calories consumed and calories burned in a given day
- **SwiftData_Store**: The local persistence layer using SwiftData for all app data
- **Navigation_Controller**: The tab-based navigation component managing screen transitions
- **Heart_Healthy_Score**: A locally-computed numeric indicator reflecting how the user's daily food and exercise choices impact cardiovascular health, based on positive factors (fiber intake, cardio workouts, staying within targets) and negative factors (heart-unhealthy flagged foods)
- **Cyan_Accent**: The accent color used for interactive elements, progress indicators, and highlighted text, a cool cyan/electric-blue-green tone (e.g., #33FFE0)
- **Sparkline**: A compact inline line chart displaying 7-day trend data for a given metric
- **Streak_Counter**: A numeric display showing consecutive days the user has met a specific daily goal

## Requirements

### Requirement 1: HealthKit Data Integration

**User Story:** As a user, I want the app to read my fitness data from Apple HealthKit, so that I can see my steps, active energy burned, workouts, and resting heart rate without manual entry.

#### Acceptance Criteria

1. WHEN the App launches for the first time, THE HealthKit_Service SHALL request read authorization for step count, active energy burned, workout data types, and resting heart rate from Apple HealthKit.
2. WHEN HealthKit authorization is granted, THE HealthKit_Service SHALL query Apple HealthKit for the current day's step count, active energy burned in kilocalories, completed workouts, and the most recent resting heart rate value.
3. WHEN HealthKit authorization is denied, THE App SHALL display a message informing the user that health data is unavailable and provide instructions to enable access in device Settings.
4. WHEN the Dashboard becomes visible, THE HealthKit_Service SHALL refresh health data from Apple HealthKit for the current day.
5. THE HealthKit_Service SHALL read workout data synced from third-party devices (including Garmin) through Apple Health without requiring direct third-party integration.

### Requirement 2: Home Dashboard Display

**User Story:** As a user, I want to see a dashboard prioritizing my key daily metrics, so that I can quickly assess my calorie balance, workout plan, activity, heart health, trends, and streaks at a glance.

#### Acceptance Criteria

1. THE Dashboard SHALL display a Progress_Ring showing the ratio of calories burned (active energy from HealthKit_Service) to the daily calorie burn goal as the primary metric at the top of the screen.
2. THE Dashboard SHALL display a card containing the current workout recommendation generated by the Workout_Recommender in a prominent position below the calories burned metric.
3. THE Dashboard SHALL display a Progress_Ring showing the ratio of calories consumed to the daily calorie intake target as the second key metric.
4. THE Dashboard SHALL display a card showing the current day's step count retrieved from the HealthKit_Service.
5. THE Dashboard SHALL display a resting heart rate card showing the current resting heart rate value in beats per minute and a 7-day trend sparkline, positioned after the step count card.
6. THE Dashboard SHALL display a section listing today's completed workouts retrieved from the HealthKit_Service, showing workout type, duration in minutes, and calories burned per workout.
7. WHEN no workouts have been recorded for the current day, THE Dashboard SHALL display an empty state message indicating no workouts logged today.
8. THE Dashboard SHALL display a Heart Healthy Score card after the workout stats section.
9. THE Dashboard SHALL display a weekly trends section containing 7-day sparklines for calories burned, calories consumed, and step count.
10. THE Dashboard SHALL display streak counters showing consecutive days the user has met the daily calorie burn goal and the daily calorie intake target.
11. THE Dashboard SHALL arrange sections in the following order from top to bottom: Calories Burned ring, Workout Recommendation, Calories Consumed ring, Steps, Resting Heart Rate, Workout Stats, Heart Health Card, Weekly Trends, Streak Counters.

### Requirement 3: Natural Language Food Logging

**User Story:** As a user, I want to log food using natural language descriptions, so that I can quickly record meals without searching through a database.

#### Acceptance Criteria

1. WHEN the user submits a natural language food description, THE Food_Logger SHALL send the text to the Food_Parser for nutritional data extraction.
2. THE Food_Parser SHALL use Apple's Foundation Models framework to parse food descriptions on-device and return structured data containing food name, estimated calories, protein in grams, carbohydrates in grams, fat in grams, fiber in grams, and a heart-unhealthy flag with a reason string.
3. WHEN the Food_Parser returns structured nutritional data, THE Food_Logger SHALL display a review screen showing the parsed food name, calories, protein, carbohydrates, and fat values before saving.
4. WHEN the user confirms the parsed food entry on the review screen, THE Food_Logger SHALL persist the entry to the SwiftData_Store with a timestamp for the current day.
5. WHEN the user rejects or edits the parsed food entry on the review screen, THE Food_Logger SHALL allow the user to modify any nutritional value before confirming.
6. IF the Food_Parser fails to parse the food description, THEN THE Food_Logger SHALL display an error message and allow the user to enter nutritional data manually.
7. THE Food_Parser SHALL use Apple's Foundation Models framework for on-device natural language food parsing without requiring network connectivity or API keys.
8. THE Food_Parser SHALL construct a structured prompt instructing the on-device model to return JSON with fields: name, calories, protein, carbohydrates, fat, fiber, heartUnhealthy, heartUnhealthyReason.

### Requirement 4: Eating Goals Tracker

**User Story:** As a user, I want to set daily targets for calories, protein, carbohydrates, and fats, so that I can monitor my nutritional intake against my goals.

#### Acceptance Criteria

1. THE Goal_Tracker SHALL allow the user to set a daily target for total calorie intake in kilocalories.
2. THE Goal_Tracker SHALL allow the user to set daily targets for protein, carbohydrates, and fat intake in grams.
3. WHEN a food entry is added to the SwiftData_Store, THE Goal_Tracker SHALL recalculate the current day's consumed totals for calories, protein, carbohydrates, and fat.
4. THE Goals/Settings screen SHALL display a Progress_Bar for each macronutrient (protein, carbohydrates, fat) showing the ratio of consumed amount to the daily target.
5. THE Goal_Tracker SHALL persist daily targets in the SwiftData_Store so targets remain available across app launches.

### Requirement 5: Local Data Persistence

**User Story:** As a user, I want all my data stored locally on my device, so that I do not need an internet connection or account to access my history.

#### Acceptance Criteria

1. THE SwiftData_Store SHALL persist all food log entries, daily nutrition totals, and user-configured goals using the SwiftData framework.
2. THE SwiftData_Store SHALL store food log entries with the following attributes: food name, calories, protein, carbohydrates, fat, and timestamp.
3. THE SwiftData_Store SHALL store daily goal configurations including calorie target, protein target, carbohydrate target, and fat target.
4. WHEN the App launches, THE SwiftData_Store SHALL load the current day's food log entries and goal configurations without requiring network connectivity.
5. THE App SHALL operate without any external database connections or remote data synchronization.
6. THE App SHALL operate without any user authentication or login flows.
7. THE App SHALL perform food parsing entirely on-device without requiring network connectivity.

### Requirement 6: Visual Design

**User Story:** As a user, I want the app to have a dark-themed design with accent colors, so that the interface is visually appealing and comfortable during nighttime use.

#### Acceptance Criteria

1. THE App SHALL use a deep cool-dark background color scheme with a blue-black undertone as the primary theme across all screens.
2. THE App SHALL use Cyan_Accent as the accent color for interactive elements, progress indicators, and highlighted text.
3. THE App SHALL apply the deep cool-dark theme and Cyan_Accent consistently across the Dashboard, Food_Logger, Goal_Tracker, and settings screens.
4. THE App SHALL use card backgrounds with a slightly blue-dark tone that complements the deep cool-dark primary background.

### Requirement 7: Tab-Based Navigation

**User Story:** As a user, I want to navigate between app sections using tabs, so that I can quickly switch between the dashboard, food log, and settings.

#### Acceptance Criteria

1. THE Navigation_Controller SHALL provide a tab bar at the bottom of the screen with distinct sections for Dashboard, Food Log, and Goals/Settings.
2. WHEN the user taps a tab, THE Navigation_Controller SHALL display the corresponding screen immediately without requiring additional navigation steps.
3. THE Navigation_Controller SHALL visually indicate the currently active tab using the Cyan_Accent color.

### Requirement 8: Workout Recommendations

**User Story:** As a user, I want to receive simple workout recommendations based on my calorie burn progress, so that I know what type of exercise to focus on.

#### Acceptance Criteria

1. WHILE the current day's active energy burned is less than the daily calorie burn goal, THE Workout_Recommender SHALL suggest cardio-focused workout activities.
2. WHILE the current day's active energy burned meets or exceeds the daily calorie burn goal, THE Workout_Recommender SHALL suggest recovery or strength-focused workout activities.
3. THE Workout_Recommender SHALL use static rule-based logic without machine learning models or external recommendation services.
4. THE Dashboard SHALL display the current workout recommendation from the Workout_Recommender in a visible card or section.

### Requirement 9: Platform and Deployment

**User Story:** As a developer, I want the project configured for iOS 26+ and physical device deployment, so that I can run and test the app on my iPhone via USB.

#### Acceptance Criteria

1. THE App SHALL target iOS 26.0 as the minimum deployment version.
2. THE App SHALL be built entirely with SwiftUI for the user interface layer.
3. THE App SHALL be structured as an Xcode project configured for deployment to a physical iPhone device via USB connection.
4. THE App SHALL compile and run without requiring any external database services or backend servers.
5. THE App SHALL require a device with Apple Intelligence support (iPhone 15 Pro or later) for on-device food parsing.

### Requirement 10: Heart Healthy Score

**User Story:** As a user, I want to see a Heart Healthy Score on my dashboard, so that I can understand how my daily food and exercise choices impact my cardiovascular health.

#### Acceptance Criteria

1. THE Dashboard SHALL display a Heart Healthy Score card as a prominent indicator on the home screen, positioned after the workout stats section.
2. THE App SHALL compute the Heart Healthy Score locally using a rule-based system that evaluates the current day's food log entries and workout data without requiring external services.
3. WHEN computing the Heart Healthy Score, THE App SHALL apply positive scoring factors for high fiber intake, completed cardio workouts from HealthKit_Service, and staying within daily calorie and fat targets defined in the Goal_Tracker.
4. WHEN computing the Heart Healthy Score, THE App SHALL apply negative scoring factors for food entries flagged as heart-unhealthy by the Food_Parser, including items high in saturated fat, excessive sodium, alcohol, and high-sugar foods.
5. WHEN a food entry is added or the HealthKit_Service refreshes workout data, THE App SHALL recalculate the Heart Healthy Score for the current day.
6. THE Dashboard SHALL display the Heart Healthy Score in a format that conveys the user's daily heart health performance at a glance.
7. THE Food_Parser SHALL return a boolean heart-unhealthy flag set to true for food items that are detrimental to heart health, accompanied by a reason string describing why the item is flagged.
8. THE Food_Parser SHALL return fiber content in grams for each parsed food entry.
9. THE Heart Healthy Score computation SHALL use only data available within the App, including food log entries from the SwiftData_Store and workout data from the HealthKit_Service.

### Requirement 11: Weekly Trend Sparklines

**User Story:** As a user, I want to see 7-day trend sparklines for my key metrics, so that I can quickly identify patterns in my fitness and nutrition habits.

#### Acceptance Criteria

1. THE Dashboard SHALL display a 7-day trend sparkline for calories burned showing daily values for the past 7 days.
2. THE Dashboard SHALL display a 7-day trend sparkline for calories consumed showing daily values for the past 7 days.
3. THE Dashboard SHALL display a 7-day trend sparkline for step count showing daily values for the past 7 days.
4. THE Sparkline charts SHALL render as compact inline line charts that fit within the dashboard layout without requiring scrolling to a separate screen.
5. WHEN fewer than 7 days of data are available, THE Sparklines SHALL display only the available days.
6. THE Sparkline data SHALL be sourced from the SwiftData_Store for food entries and from the HealthKit_Service for steps and calories burned.

### Requirement 12: Streak Counter

**User Story:** As a user, I want to see my consecutive-day streaks for hitting goals, so that I feel motivated to maintain consistency.

#### Acceptance Criteria

1. THE Dashboard SHALL display a Streak_Counter showing the number of consecutive days the user has met their daily calorie burn goal.
2. THE Dashboard SHALL display a Streak_Counter showing the number of consecutive days the user has stayed within their daily calorie intake target.
3. WHEN the user meets the goal for the current day, THE Streak_Counter SHALL increment by 1.
4. WHEN the user fails to meet the goal for a day, THE Streak_Counter SHALL reset to 0.
5. THE Streak_Counters SHALL persist across app launches using the SwiftData_Store.
6. THE Streak_Counters SHALL be computed by looking at historical daily data going backwards from today until a day is found where the goal was not met.

### Requirement 13: Resting Heart Rate Display

**User Story:** As a user, I want to see my resting heart rate on the dashboard, so that I can monitor my cardiovascular fitness over time.

#### Acceptance Criteria

1. THE HealthKit_Service SHALL request read authorization for resting heart rate data from Apple HealthKit.
2. WHEN HealthKit authorization is granted, THE HealthKit_Service SHALL query Apple HealthKit for the most recent resting heart rate value.
3. THE Dashboard SHALL display the current resting heart rate in beats per minute (bpm) in a visible card.
4. THE Dashboard SHALL display a 7-day trend sparkline for resting heart rate alongside the current value.
5. WHEN resting heart rate data is unavailable, THE Dashboard SHALL display a placeholder indicating no data available.
